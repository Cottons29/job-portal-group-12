# 6.1. Methodology: Agile Sprint Execution

The team worked in an **agile**, iterative fashion, treating each week as a short
sprint with a clear focus and a deliverable outcome.

### How sprints ran
- **Weekly cadence.** Each week targeted a coherent slice of functionality — for
  example, *auth UI & onboarding*, *backend auth & file uploads*, *profiles &
  navigation*, or *matching & discovery*.
- **Incremental delivery.** Rather than building everything up front, features
  were layered week over week, so the platform was demonstrable at many points
  along the way.
- **Front-loaded UI, then API.** In some sprints the team intentionally designed
  the UI and state ahead of the backend API, then connected the two — letting
  design feedback shape the contract.
- **Continuous refinement.** Later sprints revisited earlier work (e.g. tuning the
  matching algorithm, consolidating profiles) instead of treating features as
  "done forever".

This weekly rhythm is reflected directly in the book's
[Table of Contents](../table_of_contents.md), where each week lists its focus and
its main challenge.
